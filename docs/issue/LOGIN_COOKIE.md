## Question

그럼 현재 내 코드에서 '로그인 성공 시 access/refresh token 쿠키 저장' 은 클라이언트가 아니라
백엔드쪽에서 영향을 끼치는 역할인가? 클라이언트에서는 직접 확인할 수 없는 부분?

## Answer

네, 정확히 이해하셨습니다. 아주 중요한 포인트입니다.

결론부터 말씀드리면, 쿠키 저장은 백엔드의 역할이 맞으며, 클라이언트에서는 직접 그 내용을 확인할 수
없습니다. 이것이 바로 현대 웹 애플리케이션에서 사용하는 안전한 인증 방식의 핵심입니다.

조금 더 자세히 설명해 드리겠습니다.

1. 쿠키 저장은 누가 하는가? (백엔드의 역할)

사용자님의 로그인 흐름에서 쿠키가 설정되는 과정은 다음과 같습니다.

    1.  실제 백엔드 (Kotlin/Spring, `localhost:8080`):

        - 로그인 정보가 유효하면, accessToken과 refreshToken을 생성합니다.
        - 이 토큰들을 `Set-Cookie` 라는 HTTP 응답 헤더(Response Header)에 담아서 보냅니다.
        - 이때, 보안을 위해 중요한 옵션을 함께 설정합니다. 바로 `HttpOnly` 입니다.

    2.  Next.js 서버 (Route Handler, `localhost:3000`):

        - 실제 백엔드로부터 받은 응답을 그대로 클라이언트(브라우저)에게 전달하는 '중개자' 역할을 합니다.
        - 백엔드가 보낸 Set-Cookie 헤더를 자신의 응답에 포함시켜 브라우저로 보냅니다.

    3.  브라우저:
        - Next.js 서버로부터 받은 응답에 Set-Cookie 헤더가 있는 것을 확인합니다.
        - 헤더의 지시에 따라, 브라우저는 자동으로 토큰을 자신의 쿠키 저장소에 저장합니다.

즉, 클라이언트의 JavaScript 코드가 `document.cookie = ...` 와 같이 직접 쿠키를 저장하는 것이 아니라,
백엔드의 "이 쿠키를 저장해!" 라는 HTTP 헤더 명령에 따라 브라우저가 수동적으로 저장하는 것입니다.

2. 클라이언트에서 쿠키를 확인할 수 없는 이유 (HttpOnly)

백엔드가 쿠키를 설정할 때 HttpOnly 옵션을 사용하는 것이 핵심입니다.

- `HttpOnly` 옵션이란?

  - 이 옵션이 설정된 쿠키는 오직 HTTP(S) 통신으로만 서버에 전송될 수 있습니다.
  - 클라이언트의 JavaScript 코드 (`document.cookie`)로는 이 쿠키에 절대 접근할 수 없습니다.
    브라우저가 보안상의 이유로 접근을 원천 차단하기 때문입니다.

- 왜 이렇게 할까요? (보안)
  - 만약 JavaScript로 토큰에 접근할 수 있다면, XSS(Cross-Site Scripting) 공격에 매우 취약해집니다.
    악의적인 스크립트가 웹사이트에 주입될 경우, document.cookie를 통해 사용자의 토큰을 탈취하여 다른
    곳으로 전송할 수 있습니다.
  - HttpOnly 옵션은 이러한 토큰 탈취 공격을 막는 가장 기본적인 보안 장치입니다.

요약

| 주체         | 역할                                                                                           |
| ------------ | ---------------------------------------------------------------------------------------------- |
| 실제 백엔드  | 토큰을 생성하고, Set-Cookie 헤더에 담아 "쿠키 저장 명령"을 내림. (HttpOnly 옵션으로 보안 강화) |
| Next.js 서버 | 백엔드의 "쿠키 저장 명령"을 브라우저에 그대로 전달하는 중개자.                                 |
| 브라우저     | "쿠키 저장 명령"을 받아 자동으로 쿠키를 저장. 이후 서버에 요청을 보낼 때마다 쿠키 확인.        |
| 클라이언트   | HttpOnly 쿠키의 존재 자체를 알 수 없음. 읽기, 쓰기, 수정 모두 불가능.                          |

따라서 사용자님께서 생각하신 대로, 클라이언트는 쿠키의 존재나 내용을 직접 알 필요 없이, 그저 API를
호출하면 브라우저가 알아서 인증 토큰(쿠키)을 함께 보내주는 방식으로 동작합니다. 이것이 바로 안전하고
표준적인 웹 인증 흐름입니다.
